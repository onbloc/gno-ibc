# gnokey transaction query vectors — zkgm.Send replay

Target realm: `gno.land/r/gnoswap/ibc/v1/apps/zkgm`

The 11 scenarios below are generated by `tools/zkgm-fixtures/` from the canonical
fixture at `gno.land/p/core/ibc/zkgm/testdata/scenarios.json`. `gnokey maketx
call` cannot pass `z.Instruction` or `[32]byte` directly, so each scenario is
rendered into a small `maketx run` script via:

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh --all
# rendered files land under tools/zkgm-fixtures/scripts/out/<name>.gno
```

To execute a single scenario against a running chain, pipe `--exec` and supply
the gnokey environment:

```sh
GNOKEY_REMOTE=tcp://127.0.0.1:26657 \
GNOKEY_CHAINID=dev \
GNOKEY_KEYNAME=test1 \
tools/zkgm-fixtures/scripts/gen-send-script.sh <scenario-name> --exec
```

## gnodev local setup

zkgm packages live on disk under `gno.land/{p,r}/core/...` but declare module
names `gno.land/{p,r}/gnoswap/...` in `gnomod.toml`. The default `gnodev`
`root=` resolver matches directory layout to import path, so the aliased
modules need explicit `local=` resolvers plus pre-deployment via `-paths`:

```sh
gnodev local \
  -root "$HOME/.cache/gno-ibc/gno" \
  -resolver "root=$PWD" \
  -resolver "root=$HOME/.cache/gno-ibc/gno/examples" \
  -resolver "local=$PWD/gno.land/p/core/tokenbucket" \
  -resolver "local=$PWD/gno.land/p/core/ibc/zkgm" \
  -resolver "local=$PWD/gno.land/r/core/ibc/v1/apps/zkgm" \
  -resolver "local=$PWD/gno.land/r/core/ibc/v1/apps/zkgm/v0/impl" \
  -resolver "local=$PWD/gno.land/r/core/ibc/v1/apps/zkgm/v0/loader" \
  -paths "gno.land/r/core/ibc/v1/core,gno.land/r/core/ibc/v1/lightclients/cometbls,gno.land/r/gnoswap/ibc/v1/apps/zkgm,gno.land/r/gnoswap/ibc/v1/apps/zkgm/v0/impl,gno.land/r/gnoswap/ibc/v1/apps/zkgm/v0/loader" \
  -no-web \
  -node-rpc-listener 0.0.0.0:26657
```

## Verifying the rendered call

Every rendered script invokes:

```gno
zkgm.Send(
    cross,
    core.ChannelId(<source_channel>),
    core.Timestamp(<tx_timeout_timestamp>),
    <salt>,
    z.Instruction{Version: <version>, Opcode: <opcode>, Operand: <operand>},
)
```

The fixture carries **toy** values for `Sender`, `BaseToken`, etc., so every
scenario surfaces a realm-level validation error on the happy path. The
panic message is the **expected ground truth** for the configured fixture —
it confirms the harness reached the realm and the realm's validation chain
ran as documented. Use these as reference outputs; if a future change
causes a different panic, the diff is the regression signal.

To make a scenario succeed end-to-end, you must additionally:
- patch the rendered script's `sender` field to `runtime.OriginCaller()` (the
  gnokey signer address), and
- pre-stage realm state matching the scenario (mint vouchers, register IBC
  port, open channel, register impl version) — out of scope for fixture
  replay; the `testing/e2e/` harness owns that.

## 1. recv_call_eureka_true

Source: `scenarios.json` entry `recv_call_eureka_true`. `Call.Eureka = true`,
`source_channel = 1`, salt = `0x11…11`, operand = ABI-encoded `Call`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_call_eureka_true --exec
```

Expected:

```
panic: zkgm/v1: eureka mode not supported
ref(gno.land/r/gnoswap/ibc/v1/apps/zkgm).Send(...)
    gno.land/r/gnoswap/ibc/v1/apps/zkgm/send.gno:36
```

The v1 implementation rejects `eureka = true` at intake — eureka semantics
are not implemented in this version.

## 2. recv_call_eureka_false_empty_calldata

Source: `scenarios.json` entry `recv_call_eureka_false_empty_calldata`.
`Call.Eureka = false`, `Call.Sender = 0x616c696365` ("alice"), empty calldata.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_call_eureka_false_empty_calldata --exec
```

Expected:

```
panic: zkgm/v1: invalid call sender
```

`Call.Sender` is compared against `runtime.OriginCaller()` and rejected
because the fixture's "alice" does not match the gnokey signing address
(`g1jg8mt…` for the default `test1` keyring).

## 3. recv_token_order_v2_initialize_protocol_fill

Source: `scenarios.json` entry `recv_token_order_v2_initialize_protocol_fill`.
`TokenOrderV2.Kind = INITIALIZE`, `BaseToken = 0x756e696f6e546f6b656e` ("unionToken"),
`BaseAmount = 100`, embedded `TokenMetadata{Implementation: "grc20", Initializer: "init"}`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_token_order_v2_initialize_protocol_fill --exec
```

Expected:

```
panic: zkgm/coins: sent coin mismatch
```

INITIALIZE requires the caller to attach the originating coins via tx
`-send`. The rendered script does not attach coins, so the
`chain.NewCoins(...) vs Order.BaseAmount` check fires.

## 4. recv_token_order_v2_escrow_protocol_fill

Source: `scenarios.json` entry `recv_token_order_v2_escrow_protocol_fill`.
`TokenOrderV2.Kind = ESCROW`, `BaseToken = 0x6962632f76312d73656e64` ("ibc/v1-send").

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_token_order_v2_escrow_protocol_fill --exec
```

Expected:

```
panic: zkgm/voucher: not found: ibc/v1-send
```

ESCROW expects the base token to already exist as a voucher (i.e. a token
previously received over IBC). The fixture token `"ibc/v1-send"` is not
registered in voucher state.

## 5. recv_token_order_v2_unescrow_protocol_fill

Source: `scenarios.json` entry `recv_token_order_v2_unescrow_protocol_fill`.
`TokenOrderV2.Kind = UNESCROW`, `BaseToken = 0x756e696f6e546f6b656e`,
`BaseAmount = 500`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_token_order_v2_unescrow_protocol_fill --exec
```

Expected:

```
panic: zkgm/coins: sent coin mismatch
```

UNESCROW also requires the caller to deposit coins matching `BaseAmount`.
Same root cause as INITIALIZE.

## 6. recv_token_order_v2_solve_marketmaker_fill

Source: `scenarios.json` entry `recv_token_order_v2_solve_marketmaker_fill`.
`TokenOrderV2.Kind = SOLVE`, `BaseAmount = 1000`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_token_order_v2_solve_marketmaker_fill --exec
```

Expected:

```
panic: zkgm/v1: solve token order not implemented
```

SOLVE is not implemented in v1 — the dispatch path explicitly aborts.

## 7. recv_token_order_v1_legacy_protocol_fill

Source: `scenarios.json` entry `recv_token_order_v1_legacy_protocol_fill`.
Uses `INSTR_VERSION_1` to encode the legacy TokenOrderV1 body.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_token_order_v1_legacy_protocol_fill --exec
```

Expected:

```
panic: zkgm/v1: unsupported token order version
```

`decodeOrderInstruction` only accepts `INSTR_VERSION_2`. The legacy V1 shape
is included in the fixture for decoder-side coverage; the dispatcher rejects
it on Send.

## 8. recv_batch_empty

Source: `scenarios.json` entry `recv_batch_empty`. `Batch.Instructions = []`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_batch_empty --exec
```

Expected:

```
panic: port not found
```

An empty batch passes opcode validation but `core.PacketSend` then looks up
the source IBC port for the channel and fails because no port is registered
on a fresh `gnodev local` boot. (Non-empty Batch scenarios fail earlier
inside the sub-instructions — see scenario 9.)

## 9. recv_batch_call_then_token_order_escrow

Source: `scenarios.json` entry `recv_batch_call_then_token_order_escrow`.
Batch of `[Call{eureka=true}, TokenOrderV2{ESCROW}]`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_batch_call_then_token_order_escrow --exec
```

Expected:

```
panic: zkgm/v1: eureka mode not supported
```

The first sub-instruction is a `Call{eureka=true}`. Dispatch verifies each
sub-instruction in order and aborts on the first one.

## 10. recv_forward_single_hop_call

Source: `scenarios.json` entry `recv_forward_single_hop_call`. `Forward`
wrapping a `Call{eureka=true}`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_forward_single_hop_call --exec
```

Expected:

```
panic: zkgm/v1: eureka mode not supported
```

Forward's inner instruction is verified before the forward envelope is
accepted. The inner Call has `eureka=true` — same rejection as scenario 1.

## 11. recv_forward_recursive_two_hops_call

Source: `scenarios.json` entry `recv_forward_recursive_two_hops_call`.
`Forward → Forward → Call{eureka=false}`.

```sh
tools/zkgm-fixtures/scripts/gen-send-script.sh recv_forward_recursive_two_hops_call --exec
```

Expected:

```
panic: zkgm/v1: invalid forward instruction
```

The outer forward's inner instruction is itself a Forward, which v1 either
disallows or fails its structural check on. (Recursive forwards exist on the
wire so the decoder must handle them, but the Send-side dispatcher rejects
them — Forward chains are constructed on the destination chain, not the
source.)

## Updating this document

When `scenarios.json` is regenerated (`make refresh-zkgm-scenarios`) or
realm validation changes, re-run all 11 scenarios:

```sh
for name in $(jq -r '.[].name' gno.land/p/core/ibc/zkgm/testdata/scenarios.json); do
  echo "===> $name"
  GNOKEY_REMOTE=tcp://127.0.0.1:26657 GNOKEY_CHAINID=dev GNOKEY_KEYNAME=test1 \
    tools/zkgm-fixtures/scripts/gen-send-script.sh "$name" --exec 2>&1 \
    | grep -E '^(Data: |panic: )' | head -3
  echo
done
```

Any change in the panic line for an unchanged scenario is a behavior
regression worth investigating before updating this table.
