# State-Lens ICS23/MPT Light Client

## Context

The v1 IBC core realm needs a state-lens client that can expose an L2 IBC state
root through an L1 light client. The reference model is Union's state-lens light
client and the `state-lens-ics23-mpt` extra type:

- `lib/state-lens-light-client-types/src/client_state.rs`
- `lib/state-lens-light-client-types/src/header.rs`
- `lib/state-lens-ics23-mpt-light-client-types`
- `lib/ethereum-light-client-types/src/storage_proof.rs`

The lens client does not verify the L2 consensus state directly from an L2
header. Instead, its update proves that the L2 consensus state bytes are stored
in the L1 client state tracked by an existing client, then extracts the L2
timestamp, state root, and storage root from configured byte offsets.

## Decision

Add a v1 `state-lens/ics23/mpt` light client adapter and supporting package
types:

- encode and decode the Union-compatible client state, consensus state, and
  header shapes in `p/core/ibc/lightclients/statelensics23mpt`
- expose `core.VerifyClientMembership` so the lens adapter can verify L2
  consensus state membership through its configured L1 client without committing
  a proof result into the core realm
- make the lens client's status mirror the underlying L1 client status
- verify membership and non-membership proofs by deriving the Ethereum IBC
  commitment slot from the Union path and checking the proof against the stored
  L2 storage root
- keep the Union `StorageProof` wire codec in `p/core/ethereum/storage`, because
  it is Ethereum storage proof data rather than lens-specific state
- copy persisted consensus roots before MPT verification so cross-realm readonly
  slices do not trip Gno's taint rules inside proof verification helpers

## Alternatives Considered

Committing the L1 membership proof result through `CommitMembershipProof` during
lens updates was rejected because the lens update only needs to validate an L2
consensus state before storing the derived lens consensus state. Writing an
extra proof result into core state would add side effects that Union's lens
client does not require.

Keeping the storage proof codec inside the state-lens package was rejected
because the same `StorageProof` shape belongs to Ethereum storage verification
and is reusable by other Ethereum-backed clients or fixtures.

Using raw proof keys directly was rejected because the lens client must bind IBC
paths to Ethereum storage slots via the same `IBCCommitmentKey` derivation used
by the Ethereum storage package.

## Consequences

The v1 core now has a small verification-only helper that light clients can use
internally. The helper does not replace normal connection, channel, or packet
proof verification paths.

State-lens proofs now depend on the shared Ethereum storage proof verifier and
codec. Tests cover the codec, direct membership and non-membership verification,
inactive client rejection before proof decoding, and a core packet receive flow
that goes through the state-lens adapter.
