# Light Client P1 Storage and Initial Mirrors

## Context

The v1 Union-compatible core keeps light clients as Gno objects instead of
Union's address-based contract registry. The missing P1 behaviors still need to
match Union's side effects:

- `CreateClient` mirrors initial client and consensus state commitments.
- `VerifyCreation` and `VerifyHeader` may return client-local storage writes.
- `ForceUpdateClient` performs the same re-initialization side effects as
  client creation instead of only swapping the in-memory object.

The existing object-first API does not pass raw `MsgCreateClient` bytes into
core, so core needs a way to ask constructed light-client objects for the
initial bytes it should mirror.

## Decision

Keep the object-first API and add an optional
`lightclient.InitialStateProvider` extension. Light clients that can encode
their initial state implement:

- `InitialClientStateBytes()`
- `InitialConsensusStateBytes()`

Core calls this extension during `CreateClient` and `ForceUpdateClient`, stores
the client state, and stores the consensus state at the light client's latest
height. A non-empty `ClientCreationResult.ClientStateBytes` still overrides the
initial client state, matching Union's `verify_creation` override behavior.

Add `types.StorageWrite` and extend both `types.ClientCreationResult` and
`types.StateUpdate` with `StorageWrites`. Core stores these values in a
per-client local tree and clones values on write and read. The storage remains
internal until a concrete light client needs a public query or proof-facing
accessor.

Make `ForceUpdateClient` run `VerifyCreation`, apply initial mirrors, creation
events, client-state overrides, and storage writes, then emit a distinct
`ForceUpdateClient` event.

## Alternatives Considered

Replacing `CreateClient` with a message/bytes API closer to Union was rejected
for this step because it would force broader call-site churn and duplicate the
existing light-client constructors. The optional provider keeps compatibility
with current object-based callers while adding Union-equivalent persistence.

Adding a public client-storage read API immediately was rejected because no
production light client currently consumes it through core. Keeping it internal
reduces surface area while preserving the required state transition behavior.

Skipping `VerifyCreation` during force update was rejected because Union's
admin force update path applies creation-time side effects, including returned
events and storage writes.

## Consequences

Light clients that do not implement `InitialStateProvider` continue to work, but
core cannot mirror initial state bytes for them. CometBLS and StateLens ICS23
MPT now implement the extension.

P1 storage writes are persisted per client and covered by focused core tests,
including clone safety. The chosen design preserves the current Gno object model
while making the observable core side effects closer to Union.
