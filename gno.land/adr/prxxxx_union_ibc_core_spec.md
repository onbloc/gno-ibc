# Union IBC Core Spec Alignment

## Context

The v1 IBC core realm is intended to interoperate with Union's IBC bridge. The
authoritative reference for this change is Union's `ibc-union-spec` at commit
`6215985a5f27e24e9a34aa7259f38b0059c9a5ab`, especially `datagram.rs`,
`path.rs`, and the packet/channel/connection type definitions.

The previous core surface still exposed several Gno-specific helpers and
sequence-based packet behavior that did not exist in the Union datagram model.

## Decision

Align the exported v1 core datagrams, packet commitment paths, and event
payloads with Union:

- keep exported entrypoints to Union datagram-shaped messages plus Gno `Render`
- remove exported Gno-only packet send, raw create, app/client registration, and
  acknowledgement writer helpers from the core public surface
- represent packet commitments and receipts through Union batch packet hashes
- treat packet `timeout_height` as Union `MustBeZero`
- keep Union's empty channel-close datagrams but make them unsupported until the
  spec provides enough fields to execute them
- emit enough event attributes to reconstruct Union event semantics for
  client, connection, channel, batch, packet receive, write acknowledgement,
  packet acknowledgement, and packet timeout events

## Alternatives Considered

Keeping compatibility wrapper methods was rejected because it leaves a public
surface that relayers or apps could rely on even though Union will not produce
those datagrams.

Maintaining sequence-based packet paths was rejected because Union packet
identity is based on ABI-encoded packet batch commitments, not ICS-04 sequence
paths.

## Consequences

Apps and light client adapters may need follow-up changes to use the narrowed
core surface. Event type strings keep the existing Gno style, while event
payloads are aligned by meaning and field count with Union.
