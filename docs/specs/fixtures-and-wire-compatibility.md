# Fixtures and Wire Compatibility Spec

This document describes the fixture and compatibility surfaces used to keep
Gno-side IBC and ZKGM behavior byte-compatible with external systems.

## ABI Ground Truth

Solidity ABI compatibility fixtures are generated by the Rust harness under
[tools/abi-fixtures](../../tools/abi-fixtures). The committed Gno fixture copy
is stored at
[gno.land/p/core/encoding/abi/testdata/vectors.json](../../gno.land/p/core/encoding/abi/testdata/vectors.json).

Regenerate ABI vectors with:

```sh
make refresh-abi-vectors
```

Tests should assert both directions:

- Gno encode output equals the fixture hex
- Gno decode output reconstructs the expected fields

ZKGM wire bytes use the ABI params encoding flavor. Do not use plain
top-level `abi.encode` for packet, acknowledgement, or instruction vectors.

## ZKGM Scenario Fixtures

End-to-end ZKGM instruction and handler scenarios are generated by
[tools/zkgm-fixtures](../../tools/zkgm-fixtures). The committed scenario copy
is stored under the ZKGM package testdata and exercised by Gno tests.

Regenerate scenarios with:

```sh
make refresh-zkgm-scenarios
```

Sender-salt bootstrap vectors can be printed with:

```sh
make derive-sender-salt-vectors
```

Fixture changes should include both the harness change and regenerated Gno-side
fixture output.

## Cross-Chain Smoke and Proof Fixtures

Runtime smoke helpers live under
[tools/gnokey-smoke](../../tools/gnokey-smoke), with top-level wrappers such as
[tools/gnokey-query-smoke.sh](../../tools/gnokey-query-smoke.sh). The Makefile
exposes these through targets including `test-gnokey-query-smoke` and
`test-zkgm-native-refund-smoke`.

Ethereum storage proof fixture generation lives under
[tools/gen-ethereum-storage-proof-fixture](../../tools/gen-ethereum-storage-proof-fixture).

These smoke and proof fixtures are operational compatibility tests, not
canonical protocol specifications. When fixture behavior diverges from
implementation specs, reconcile the source implementation and fixture generator
before changing expected outputs.

## Event and Binary Data Rules

Event attributes that carry wire data must preserve binary compatibility. Use
typed encoding or explicit hex/base64-style representations where appropriate;
do not rely on lossy printable string conversions for arbitrary packet,
acknowledgement, proof, or commitment bytes.

Compatibility tests should prefer fixture-backed byte assertions over visual
inspection of event strings.

## Maintenance Rules

- Add new external compatibility cases in the generator first, then regenerate
  committed fixtures.
- Keep fixture names stable enough for downstream debugging.
- Commit generator source changes together with regenerated fixture outputs.
- Treat binary fixtures as byte data even when the file format is JSON or text.

## Maintenance Notes

This spec should track committed fixture generators, committed fixture data, and
current compatibility rules only. Keep historical planning notes out of
committed implementation specs.
